# AI Clean Comments

Remove AI-generated comment noise from code files without touching business logic or useful documentation.

## What to Remove

### Redundant comments (repeat the next line)
Comments that describe exactly what the immediately following line of code does, adding no new information:
```typescript
// increment counter          ← remove this
counter++;

// return the result          ← remove this
return result;

// call the API               ← remove this
await fetchUser(id);
```

### Template / boilerplate phrases
Comments containing these patterns (case-insensitive):
- "This function is responsible for…"
- "This method is responsible for…"
- "In this code we will…"
- "The following function…"
- "The purpose of this function is…"
- "This component renders…" (when followed by an obvious component name)
- "Helper function that…" (when the function name already says it)

### AI meta-phrases
- "As an AI…"
- "Note: this was generated by AI"
- "AI-generated"
- "Generated by Claude / ChatGPT / Copilot"
- "Note: I cannot…"

### Debug-labeled comments
- `// debug`
- `// temp` / `// temporary`
- `// temporary test`
- `// TODO remove` (standalone, with no context about what to remove or why)
- `// test` (when it's a debug marker, not a test file annotation)
- `// remove before commit`
- `// just for testing`

### Inflated JSDoc blocks on trivial functions
A JSDoc block is "inflated" when:
- The function body is 1–3 lines
- The JSDoc has more than 5 lines
- The `@description` or summary repeats the function name verbatim
- There are no `@param` or `@returns` that add real type/constraint info

Example to remove:
```typescript
/**
 * getUserById
 *
 * This function is responsible for retrieving a user by their ID.
 * It takes an ID parameter and returns the corresponding user object.
 * This is a helper function used throughout the application.
 *
 * @param id - the id
 * @returns the user
 */
function getUserById(id: string) {
  return users.find(u => u.id === id);
}
```

## What to Preserve

### Maintenance markers
Always keep these prefixes intact, regardless of surrounding content:
- `TODO:`
- `FIXME:`
- `HACK:`
- `NOTE:`
- `XXX:`
- `BUG:`
- `WORKAROUND:`

### Tool directives (never touch these)
- `eslint-disable` / `eslint-disable-next-line` / `eslint-enable`
- `@ts-ignore` / `@ts-expect-error` / `@ts-nocheck`
- `prettier-ignore`
- `biome-ignore`
- `stylelint-disable`
- `/* c8 ignore */` / `/* istanbul ignore */`
- Any comment containing `noinspection` (JetBrains IDEs)

### Valuable explanatory comments
Keep comments that explain:
- **Business rules** that aren't obvious from the code ("Fee waived for users with plan > Pro")
- **Edge cases** and why special handling is needed
- **Design decisions** and the rejected alternatives
- **Workarounds** for external bugs or platform quirks
- **Invariants** that must hold but aren't enforced by types

### JSDoc on public API surfaces
Keep `@param`, `@returns`, `@throws`, `@deprecated`, `@since` on:
- Exported functions and classes
- Public methods in exported classes
- Functions in `.d.ts` files

## Dry-Run Mode

If the user's request includes `--dry-run`, `dry run`, or "muéstrame qué cambiaría" / "show me what would change":

1. Run the full procedure below, but **do not write any file changes**.
2. Instead, output a unified diff showing what would be removed:
   ```diff
   --- src/utils/format.ts (before)
   +++ src/utils/format.ts (after)
   @@ -10,7 +10,4 @@
   -// This function is responsible for formatting the user's name
    function formatUserName(user: User): string {
   -  // concatenate first and last name
      return `${user.firstName} ${user.lastName}`;
    }
   ```
3. Print a summary (X comments would be removed) and ask the user to confirm before proceeding.
4. If the user confirms, run again in normal mode to apply changes.

## Configuration

If a `.ai-slop-cleaner.json` file exists in the project root, read it and apply `rules` and `perDirectory` overrides before processing each file. Directory-level rules take precedence over top-level rules for files within that directory.

## Procedure

1. Read the full file content.
2. Check if a `.ai-slop-cleaner.json` config exists and load relevant rules for this file's directory.
3. Identify all comment blocks (single-line `//`, multi-line `/* */`, JSDoc `/** */`).
4. For each comment, check in order:
   a. Is it a tool directive? → **Keep unconditionally.**
   b. Does it start with `TODO:`, `FIXME:`, `HACK:`, `NOTE:`, `XXX:`, `BUG:`? → **Keep unconditionally.**
   c. Does it explain a non-obvious business rule, edge case, or design decision? → **Keep.**
   d. Does it match any "What to Remove" pattern? → **Remove.**
   e. Is it a JSDoc on an exported function/class? → **Keep only if it adds info beyond the name.**
   f. If uncertain, **keep the comment** — the goal is to remove obvious noise, not to second-guess every comment.
5. Remove only the identified comments. Do not reformat or modify surrounding code.
6. If removing a comment leaves a blank line that creates awkward double-blank-lines, collapse to a single blank line.
7. Report a summary: how many comments were removed and of which type.

## Edge Cases

- **Inline comments at end of line** (`counter++; // increment`): remove the comment, keep the code line.
- **Section dividers** (`// ─────────────────`): keep if they visually separate distinct sections; remove if they're random decoration.
- **Commented-out code blocks**: do NOT remove unless explicitly asked. Commented-out code is not the same as a comment about code.
- **License headers** at the top of files: always keep.
- **File-level `@fileoverview` or `@module` JSDoc**: keep if it names the module purpose clearly; remove if it just repeats the filename.
